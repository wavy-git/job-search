## Operating model

### Three-touch rule

Every application gets three parallel touches:
1. Tailored resume submitted via ATS
2. Hiring manager or VP direct message on LinkedIn
3. Referral hunt (1st/2nd degree connection at the company)

No more pure cold-apply at Director level.

### Fresh tailor per role

Every role gets a resume tailored from the master. No generic submissions. No recycled versions between unrelated roles.

### Volume target

5-10 serious tailored applications per week with three-touch approach. Better than 40+ untailored cold apps.

## Rubric 1: Candidate Fit Screening

### Purpose

Run the full JD against Maz's full career, qualifications, and background. Answer: how strongly does Maz match this role?

This is the same evaluation an AI screener, recruiter, or hiring manager performs. Rubric 1 produces an honest, comprehensive read.

It is independent from Rubric 2. Rubric 1 = candidate-fit / compatibility scoring. Rubric 2 = execution and gap-closing when applying.

Rubric 1 is intentionally NOT mechanical-checklist scoring. The structure below is a guide to ensure nothing is missed; the screen itself should feel like a senior career coach or recruiter reviewing one last time. Less rubric-as-formula, more expert read.

### Hard gates (binary; fail any → entry goes directly to archive.md, never enters candidates.md)

The URL is logged in archive.md with the hard-gate failure reason in Notes, so future daily runs do not re-screen the same role.


- **Title keyword match.** Title must contain at least one of: product, head of product, vp product, chief product, cpo, strategy, strategic initiatives, chief strategy, cso, operations, ops, operating officer, coo, chief operating, transformation, business transformation, digital transformation, innovation, chief innovation, platform, technical platform, platform engineering, program management, program manager, project management, pmo, project management office, process excellence, process improvement, business process, continuous improvement, lean, delivery, director of delivery, head of delivery, implementation, head of implementation, general manager, gm, head of business, business head, chief of staff. AND must NOT contain: engineer, engineering manager, tech lead, technical lead, architect, developer, designer, scientist.
- **Location.** Pass if the role's location matches any of: Remote, Remote US, Philadelphia, Greater Philadelphia, Greater Philly, Philly, NYC, New York, Seattle, SF Bay, San Francisco, Los Angeles, LA, Portugal, Lisbon, Spain, Madrid, Barcelona. Greater Philly area also passes: includes any New Jersey city (Marlton, Egg Harbor, Cherry Hill, Hoboken, etc.), Wilmington Delaware, Radnor PA, Fort Washington PA, Newark DE, and other suburbs within ~60 miles of Philadelphia. Lenient default for ambiguous US labels: if location reads "United States" or "USA" without an explicit remote/onsite indicator, pass through. Unknown location passes through.
- **Posted recency.** Within last 7 days. Use the JD's posted_date if available, otherwise the email receipt date for LinkedIn alerts. Unknown posted date passes through.
- **Citizenship knock-out.** Drop if the role is in a non-US country that requires local citizenship and will not sponsor.
- **Comp floor.** If comp is listed and below floor (US Director $200K base; Portugal/Spain Director €120-140K base), drop.
- **Hard knock-out keyword.** Drop if the JD has an explicit must-have Maz cannot honestly claim: "PMP REQUIRED," "active Secret clearance REQUIRED," "biotech/pharma R&D experience REQUIRED," "5+ years recent direct PM management REQUIRED" combined with no flexibility language, or any single specialty Maz cannot defensibly bridge into.

### Screening dimensions (used with expert judgment, not rigid scoring)

Run each JD through these lenses:

1. **Pillar match map** — which of the 11 background pillars activate, and how strongly (direct / adjacent / stretch).
2. **Altitude alignment** — JD altitude (Manager / Sr Manager / Principal / Director / Head / Sr Director / VP / MD) against Maz's payroll-safe range. Company size affects how the same title reads:

  | Title | S (<500) | M (500-5K) | L (5K+) |
  |---|---|---|---|
  | Manager / Sr Manager / Principal | 0 | 0 | 0 |
  | Director / Head | 0 | 0 | -2 |
  | Sr Director | -1 | -1 | -3 |
  | VP / MD | -1 | -2 | -3 |

3. **Recent-direct-reports gap** — Maz's last direct people management ended 2015 (Infotech VP: 12 direct reports, 50+ employees, 700+ field technicians). JDs requiring "currently managing X direct reports" or "5+ years recent direct PM management" carry friction. Surface as a flag; don't auto-drop unless paired with no flexibility language.
4. **Domain bridging** — direct fit / adjacent / cross-bridge / no defensible bridge. Bridge framings live in the domain-bridging playbook (Rubric 2).
5. **Friction signals** — recruiter-Boolean filter risks (specialty domain keywords like NetSuite/Q2C/MSSP/MDR/biotech/Oracle ERP/IWMS, certification requirements, FAANG/fintech-pipeline pedigree expectations, hidden hard requirements buried in "preferred").
6. **Function fit** — Product / Strategy / Operations / Transformation / Platform / Program Management / Innovation / Process are wanted. Marketing / Design / Legal / Engineering Management / Customer Success are wrong function, drop.
7. **Realistic story** — what story would Maz tell to land this role? If no defensible story exists, the fit is weaker than the surface signals suggest.

### Output per JD

- **Score** (1-10): single composite reflecting overall match strength. Score must honestly reflect ALL factors, including knock-outs and friction signals. A role with strong pillar fit but a hard knock-out gets a low score, not a high score with a flag. Top of list = next to apply; bottom of list = consider archiving.
- **Notes** (1-2 sentences): honest read capturing the dominant fit signal and the dominant friction signal. Voice of a career coach, not bullet-list output.
- **Status field** (existing column) is set to `screened` after intake.

There is no separate Verdict column. Score is the verdict; Status drives action state.

### Channel-agnostic input

Whatever the source, Rubric 1 takes the same input shape: URL, title, company, location, posted date, JD body. Source becomes a tag (`linkedin`, `scanner`, `manual`, `recruiter-fronted`), not a branch in the logic.

Sources currently feeding the pipeline:
- Daily LinkedIn alert digest (5:00 AM ET scheduled task; collect-only)
- Python scanner output across 108 target companies (collect-only)
- Opportunistic JDs the user pastes
- Recruiter-fronted hidden-employer postings (data limited until reveal; score with available info)

### Two screens, one purpose

The same JD-vs-candidate evaluation runs at two moments with different depth:

- **Intake screen** (every JD that clears hard gates): comprehensive but fast. Produces Score + Notes. Writes to candidates.md with Status = `screened`.
- **Apply-time screen** (when we decide to pursue a role): expert-like deep validation pass, last-check-before-submit. Re-runs the full JD against the full career. Catches anything intake missed. If the score shifts materially, the row gets updated.

The apply-time screen is the natural calibration check on the intake screen. If apply-time consistently surfaces gaps intake missed, intake calibration needs adjustment.

### candidates.md schema

| Rank | Score | URL | Company | Title | Size | Source | Posted | First Seen | Last Seen | Status | Notes |

Sort order:
1. Status: `on-deck` first (active work floats to top), then `screened`, then `archive-review` (sinks to bottom for user-confirmed archival).
2. Score descending within each Status group.

`archive-review` is a Status used when the screener thinks the role should go to archive.md but wants user confirmation first. After confirmation, the row moves to archive.md.

### Status flow

- `screened` — intake screen complete; default state for any row in candidates.md.
- `on-deck` — actively pursuing (resume tailoring or outreach in progress).
- `archive-review` — flagged during screening for archive; awaiting user confirmation. After confirmation, row moves to archive.md.
- `applied` — submitted (row moves to applications.md).

Status is the single source of truth for what action a row is awaiting; there is no separate Verdict layer.

### What's explicitly OUT of Rubric 1

- Funnel-stage probability splits (recruiter-pass / HM-pass / end-to-end). Apply-phase concerns; live in Rubric 2 reasoning if needed at all.
- Warm-path data (1st/2nd LinkedIn connections at the company). Apply-phase factor.
- Mission/passion fit. Subjective leverage at apply / interview phase, not a qualification factor. JD-vs-candidate fit is objective.
- Company stage / size as a fit multiplier. Company size already affects altitude (dimension 2) but does not change Maz's underlying qualifications.
- Confidence tags for hidden-employer roles. Score with available info; flag for reveal in Notes if relevant.

### Daily intake operational pattern

1. Daily channel runs produce snapshots (LinkedIn alerts at 5:00 AM ET; Python scanner when run; opportunistic JD pastes ad-hoc).
2. New URLs (not already in candidates.md, applications.md, or archive.md) clear hard gates.
3. Hard-gate failures are logged directly to archive.md with the failure reason. They never enter candidates.md.
4. Survivors batch-processed by a subagent that fetches each canonical JD and runs intake screen.
5. Subagent writes results into candidates.md with Status = `screened`. Existing URL `last_seen` timestamps updated.
6. The same subagent pattern handles every channel; volume (not source) determines whether to batch through subagent vs run inline.

### History handling

Each daily run is independent. Repeats from previous days are fine; existing URLs only get `last_seen` updated, not re-scored.

## Rubric 2: Resume Tailoring (5-phase framework)

Used per role being applied to. The two screens we must clear:
1. **AI/ATS screen** (automated parsing + scoring against the JD before any human sees the resume)
2. **Recruiter 6-second scan** (human eyepath across page 1 to decide click-into vs discard)

Both must pass or no interview.

### Modern ATS reality (validated 2026)

- Modern ATS (Workday, Greenhouse, Ashby, Lever, plus AI-screening layers like Eightfold/HireVue) generate vector embeddings of resume + JD and compute semantic similarity — not exact keyword matching. ~78% of initial screenings now use NLP-based semantic matching.
- Keyword stuffing is actively penalized — repeating terms unnaturally lowers the score.
- What wins: contextual demonstration of skills through achievement bullets, not labels in a list.
- Knock-out filters (location, citizenship, YOE, direct-reports) fire before the AI score is even calculated. Missing a knock-out = automatic reject.
- Older parsers still scan skim zones for exact-string keywords. Recruiter Boolean searches still hit on exact strings.

### Skim zones vs narrative zones (the core conflict resolution)

The conflict between ATS keyword surface and 6-second scan readability is resolved by zone:

- **Narrative zones (Summary, role scope lines, bullets):** clean prose, achievement-driven, concrete numbers, max 1-2 keywords per natural construction. Demonstration over labeling. NEVER stuff keywords into these.
- **Skim zones (Core Competencies, Tools & Technologies, Certifications):** carry exact JD keyword surface. Stuffing acceptable here. Recruiter Boolean and older ATS parsers scan these.

This zone discipline alone resolves most ATS-vs-scan trade-offs.

### Phase 1: JD reconnaissance (15 min)

- Pull JD body from canonical source (company ATS like Ashby/Greenhouse, NOT LinkedIn abbreviations).
- Extract:
  - **Must-haves:** hard required qualifications (years, skills, scope, domain)
  - **Nice-to-haves:** preferred qualifications
  - **Knock-outs:** filters that auto-reject (location, citizenship, recent direct-reports, hard YOE bar)
  - **Culture cues:** founder language, mission framing, brand vibe
- Detect company stage: Series A (≤100 ppl) / Mid-stage SaaS (100-1K) / F500 (5K+) / FAANG.
- Detect application volume signal (45 applicants vs 200+; affects the human-review-vs-ATS-aggression weighting).
- Map each must-have to the 11 pillars 1:1 where possible.
- Flag must-haves with **no pillar match** — these are the gaps that determine pursue-or-skip judgment.

### Phase 2: Tailor flavor selection (5 min)

Pick by stage:

| Stage | Optimize for | What dominates |
|---|---|---|
| Series A (≤100 ppl) | Founder review, narrative quality, mission fit, builder identity | Cover letter + summary punch + Independent Ventures lead with Jingle/Lighthouse working apps |
| Mid-stage SaaS (100-1K) | Hiring manager review, platform thinking, agile delivery, KPIs | Bullets + Core Competencies with agile/B2B SaaS keywords |
| F500 corporate (5K+) | ATS keyword surface + recruiter scan, scale signals, regulated-industry credibility | Skim zones heavy; narrative tight |
| FAANG / Big Tech | Bar-raiser interview prep, principles match (Amazon LP / Meta XFN), data rigor | Bullets must mirror the company's leadership principles, technical depth surfaced |

### Phase 3: Tailor (30 min)

Apply tailor flavor across sections:

1. **Title alignment:** match JD's title verbatim where defensible (functional descriptor, no payroll inflation). "Head of Products" → "Head of Product, AMI Platform" for a "Head of Product" JD.
2. **Summary:** 3-4 clean sentences, hard numbers, max 1-2 keywords per natural construction. Brand line at end intact. NO keyword stuffing.
3. **Scope lines:** tight, JD-relevant lens, same facts. Avoid kitchen-sink keyword piles.
4. **Bullets:** reorder, replace, or promote interview-only material per the promotion rule below. Every modified bullet must pass the 10-point bullet framework.
5. **Independent Ventures order:** Jingle/Lighthouse first depends on JD signal — Jingle first when JD wants player-coach / hands-on / live builder; Lighthouse first when JD wants operational visibility / risk / governance / PMO.
6. **Core Competencies (skim zone):** stuff with JD's exact must-have keywords. Drop master entries that don't fit, add JD-aligned slots.
7. **Tools & Technologies (skim zone):** reorder to lead with JD-relevant tools. Stuffing acceptable here.
8. **Certifications (skim zone):** reorder to lead with JD-relevant certs.

#### Hard locks (never change, never claim what's not real)

- **Hard numbers:** 10K to 120K, 150M+ transactions, 30+ FIs, $1.4B program, 99.5/100, 12 direct reports, 50+ employees, 700+ field technicians, 1M+ terminals, $250M+, 2M+ vouchers monthly, 200% throughput, 6-8 weeks to 3-5 days, 5-6 months to 2-3 weeks, 2 to 8 FIs, 10K to 100K+ terminals, 60K terminal contract, 20 to 5,000+ NCC, 50% market share.
- **Companies, dates, employment facts.** Chronological structure.
- **Payroll-title risk:** never claim "Director" as current NG title (background-check risk). Functional descriptors only.
- **Pillar identity:** the tailored resume may reframe pillars in JD-relevant language but never claims a pillar Maz doesn't have. Don't claim agentic AI track record (it's AI-augmented automation, not agentic), don't claim healthcare domain (regulated-industry transferable framing only), don't claim waste-industry or telehealth domain expertise.

#### Promotion rule for interview-only material

The comprehensive doc lists material kept "interview-only, not on resume" (CCRE change management, AI-augmented PowerShell+Jira automation, L+G deployment gap analysis, ServiceNow migration). Promotion to resume bullets requires:

(a) Strengthens the resume's match to the JD's must-haves OR nice-to-haves
(b) Passes the 10-point bullet framework (hard outcome, scannable, executive tone, defensible)
(c) Doesn't displace a stronger bullet in the master that maps to a must-have for THIS specific JD (displacement test)

**Interview-only material status table:**

| Material | Hard outcome? | Defensible? | Promotable per rule? |
|---|---|---|---|

#### Domain-bridging playbook

When Maz's pillars don't directly match the JD's industry, use these defensible framings (no fabrication):

| Target industry | Bridge framing | Justification |
|---|---|---|
| Marketplace SaaS (Sourgum, etc.) | "B2B SaaS marketplace" | Infotech B2C PSP serving merchants is a multi-sided platform; multi-bank terminal is consolidation in fragmented market |
| Telehealth / healthcare ops (Wheel, etc.) | "Regulated infrastructure at national scale" | NG is regulated; HIPAA isn't claimed but regulatory operating muscle transfers |
| Energy / commodities exchange (ElectronX, etc.) | "Regulated payments + energy intersection" | NG energy + Infotech payments + central bank seat |
| Cybersecurity SaaS (Cobalt, Microsoft Defender, etc.) | "Operational intelligence in cybersecurity" — Comcast Log4Shell pattern, NOT MDR/MSSP product claim | Adjacent expertise only; don't claim MSSP/MDR specialty |
| AI/automation product (Wheel, Ladders client, etc.) | "AI-augmented operations + automation builder" | NG PowerShell+Jira automation, Jingle stack; do NOT claim agentic AI track record |

### Phase 4: QA gate (15 min, ship-or-don't)

Resume-level checks. Run all in order. If all pass, ship. Do not iterate further on phrasing.

1. **6-second mock scan:** read only the top of page 1 (name, title, current company, top 3 bullets) in 6 seconds. Can you state the punchline?
2. **JD must-have coverage check:** every must-have demonstrated somewhere (bullet, scope line, or skim zone)
3. **Knock-out check:** location, citizenship, YOE, direct-reports — all met
4. **10-point bullet framework on every modified bullet** (existing per-bullet check)
5. **Length:** 1-2 pages
6. **Trim 10%:** every word earns its place
7. **Side-by-side diff with reasoning** (master vs tailored): present to user for review/feedback using the canonical format below. No exceptions, no font experiments — the format below is locked because it's the only one that renders correctly in Maz's Claude Code desktop renderer.

   **Canonical side-by-side format (locked 2026-04-27):**
   - **Removed text:** `~~***[...]***~~` (strikethrough + bold-italic + square brackets)
   - **Added text:** `***[...]***` (bold-italic + square brackets)
   - **Table structure:** 5 columns — `# | Section | Master | [Tailored Company] | Why`
   - **Cell content:** show only the sections that changed; drop unchanged rows. Within each cell, show enough surrounding context for the reader to locate the change, but trim aggressively (`...` is fine for unchanged middle).
   - **Do not use:** monospace/code style (changes font, dilutes signal); HTML tags like `<u>`, `<mark>`, `<span>` (Maz's renderer strips them and shows literal text); yellow background (renderer doesn't support background colors). Bold alone is too subtle; ALL CAPS is ugly.

   Example row demonstrating the format:

   `| 1 | Summary | ...stalled ~~***[fintech startup]***~~; scaled it from 10K... | ...stalled ***[payments venture]***; scaled it from 10K... | Generalizes industry framing for hidden employer. |`
8. **Apply user feedback if any.**
9. **SHIP.** Do not iterate further on phrasing after this gate.

### 10-point bullet framework (per-bullet, used in Phase 3 and Phase 4)

Every modified bullet must pass:
1. **Outcome first** — leads with what changed
2. **Scale signal** — shows the size of impact
3. **So what** — recruiter would care
4. **One idea** — says one thing clearly
5. **Scannable** — 1.5 lines max
6. **Executive tone** — sounds like a leader wrote it
7. **Decision-maker signal** — shows Maz made a choice
8. **Altitude** — right level of abstraction
9. **Defensible** — can be backed up in interview
10. **Pattern reinforcement** — supports career thesis

## Screening (manual, by human reviewer)

Both the scanner and the LinkedIn alert task are collect-only. Rubric 1 is applied manually when the snapshot files are reviewed. The reviewer:
1. Opens `outputs/scanner/jobs_YYYY-MM-DD.md` and `outputs/linkedin/linkedin_alerts_YYYY-MM-DD.md`
2. Checks each role against `candidates.md` to dedupe and update `last_seen` for known URLs
3. Adds new URLs to `candidates.md` with `first_seen = today, status = new`
4. Applies Rubric 1 with judgment, scoring with the signals above and weighing screening inputs
5. Updates each candidate's status (`new` → `screened` → `on-deck` → `applied`/`rejected`)
6. When a candidate becomes `applied`, adds a row to `applications.md` and any outreach to `outreach.md`

## Working rules

### Memory and file hygiene

- **Update memory files at each step** based on progress and relevancy. This includes this strategy doc, `candidates.md`, `applications.md`, `outreach.md`, `archive.md`, `target-companies.md`, `interview-prep.md`, `weekly-metrics.md`, and the comprehensive career history (`Maz_Career_Background_Comprehensive.md`) in the project folder.
- **Pipeline file separation (no duplication)**:
  - `candidates.md` = active pre-application pipeline. Statuses: `screened`, `on-deck`. Two parallel lists (Realistic Baby Step, Realistic Stretched).
  - `applications.md` = submitted applications and downstream funnel. Statuses: `submitted`, `recruiter-screen`, `phone-screen`, `hm-conversation`, `interview-loop`, `onsite-loop`, `offer`, `rejected-by-employer`, `withdrawn`, `ghosted`.
  - `archive.md` = closed-out records. Statuses: `rejected`, `stale`. Kept so we don't re-evaluate the same role next time it appears.
  - `outreach.md` = every outreach touch keyed to a Company / Job URL.
  Transitions: row moves between files (no duplication). Submit → out of candidates, into applications. Reject or stale → out of candidates, into archive.
- **When the user shares new career details** not already in `Maz_Career_Background_Comprehensive.md`, update it.
- **Run a daily hygiene review** across all memory files. Comprehensively analyze for inconsistencies, stale data, missing context, and fix them. Report what changed.
